# ADR 0003: Doppler Secrets Structure

## Status
Accepted

## Context
Upstream AI provider API keys (OpenAI keys, Anthropic keys, AWS credentials) are sensitive credentials. If we store these keys in plaintext in the SQLite control-plane database, or pass them in raw environment variables, we risk credential leakage during node compromises, backup leaks, or inadvertent logging.
At the same time, we need to support environment scoping (`tst`, `stg`, `prd`) and node-level least-privilege scoping so that a compromised region or node cannot access secrets for another region.

## Decision
We will use **Doppler** as the sole source of truth for runtime secrets:
1.  **Secret Reference Mapping**: The control plane registry will store *references* to secrets (e.g., `doppler://PROJECT/CONFIG/SECRET_NAME`) instead of the actual key values.
2.  **Config-Scoped Service Tokens**: Each LiteLLM node or group of nodes will be assigned a Doppler service token scoped to its specific environment branch and config path.
3.  **In-Memory Resolution**: The Config Generator will resolve these references using the Doppler API during configuration generation, injecting them directly into the output configuration file, which is written to a memory-mapped drive (or securely permissioned local path) on the target node.
4.  **No Raw Secrets in DB or Logs**: Database audits and API responses must strip or mask resolved values.

## Consequences
*   **Reduced Attack Surface**: No API keys are written to the control plane's permanent database or backups.
*   **Rotation**: Rotating a secret in Doppler automatically updates the next configuration generated by the control plane without requiring database migrations.
*   **Dependency**: Config generation depends on Doppler API availability. We must implement caching or fallbacks to keep generating configs using last-known-good cached secret tokens in case of transient Doppler outages.

## Alternatives Considered
*   *Control Plane Vault*: Storing keys encrypted in SQLite/Postgres. Rejected due to the operational complexity of managing encryption keys, backup safety, and audit logs.
*   *Plaintext Env Files*: Storing `.env` files on each node manually. Rejected because it makes rotation and automated configuration rollouts impossible.
